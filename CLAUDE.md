## Commands

- `npm run dev` — dev server at http://localhost:3000
- `npm run build` — production build
- `npm run lint` — ESLint
- `psql "$DATABASE_URL" -f db/setup.sql` — create app tables (idempotent)
- `npx @better-auth/cli@latest migrate --config lib/auth.ts` — create/update better-auth-owned tables (needed once for the mcp plugin's `oauthApplication`/`oauthAccessToken`/`oauthConsent`; `generate` first to review the SQL)
- No test suite exists.

Local Stripe webhooks: `stripe listen --forward-to localhost:3000/api/auth/stripe/webhook` (its signing secret goes in `STRIPE_WEBHOOK_SECRET`). Required env vars are documented in `.env.example`.

Scheduled workflow execution is triggered externally — a cron-job.org job (manual dashboard setup, not in the repo; no vercel.json) pings `GET /api/cron` every minute with `Authorization: Bearer $CRON_SECRET`.

## Stack

Next.js 16 App Router (with `experimental.viewTransition`), React 19, Tailwind CSS 4 (no config file — theme, view-transition keyframes, and tier glow utilities live in `app/globals.css`), TypeScript, better-auth + `@better-auth/stripe`, raw `pg` Pool against Neon Postgres (no ORM). Fonts come from the `geist` package + Google Inter, loaded in `app/layout.tsx`.

`next.config.ts` sets global security headers via `headers()` (applied to `/:path*`): a CSP (`default-src 'self'`; `script-src`/`style-src` allow `'unsafe-inline'` for Next's inline bootstrap + the designer's inline styles; `img-src` allows `data:` + Google s2 favicons + the avatar host; dev additionally allows `'unsafe-eval'` + `ws:` for Fast Refresh/HMR — computed inside `headers()` so `NODE_ENV` is resolved at bake time), plus `frame-ancestors 'none'`/`X-Frame-Options: DENY` (clickjacking, notably the OAuth consent page), HSTS, `nosniff`, `Referrer-Policy`, and a minimal `Permissions-Policy`.

## Architecture

### Auth + billing (the core of the app)

Everything flows through better-auth, configured in `lib/auth.ts`:

- Google OAuth is the only sign-in method.
- `app/api/auth/[...all]/route.ts` is the single catch-all handler for all auth routes **and** the Stripe webhook (`/api/auth/stripe/webhook`) — the stripe plugin handles webhooks; don't add a separate webhook route. The only other routes under `app/api/` are the MCP OAuth callback (`app/api/mcp/oauth/callback/route.ts`) and the cron tick (`app/api/cron/route.ts` — constant-time bearer check against `CRON_SECRET`, 401 on mismatch, 500 when the secret is unset so it never operates open; calls `runDueWorkflows`). Outside `app/api/`: the hosted MCP server at `app/mcp/route.ts` and the root OAuth discovery documents (`app/.well-known/oauth-authorization-server/` + `oauth-protected-resource/` and its `/mcp`-suffixed form) — see "Hosted MCP server" below.
- The `mcp` plugin (better-auth built-in) makes the catch-all an OAuth 2.1 authorization server (dynamic client registration + PKCE at `/api/auth/mcp/authorize|token|register`) for the hosted MCP server; `loginPage: "/onboard"` — an unauthenticated authorize request bounces through Google sign-in and resumes automatically. **Consent is forced** (fixes a cross-account-takeover: the plugin issues a code with no consent unless the request carries `prompt=consent`, which — with anonymous client registration + a Lax session cookie — let a logged-in victim be lured to an authorize URL that silently minted a token for an attacker's client). Root-level **`proxy.ts`** (Next 16's renamed middleware; matcher `/api/auth/mcp/authorize`) rewrites every authorize request to add `prompt=consent`, and `mcp({ oidcConfig: { consentPage: "/oauth/consent" } })` routes it to the first-party consent screen `app/oauth/consent/page.tsx` (server component; Approve/Deny posts through a server action to `auth.api.oAuthConsent`, then redirects to the client's callback). Do not remove the proxy or `consentPage` without replacing the consent gate. The post-login resume path is covered because the plugin replays the already-rewritten query from its signed cookie. Its tables (`oauthApplication`, `oauthAccessToken`, `oauthConsent`) are better-auth-owned — migrate via the CLI, never `db/setup.sql`.
- The `nextCookies` plugin must stay last in the plugins array.
- `databaseHooks.user.create.after` seeds every new user an inactive "Daily haiku" example workflow (`seedExampleWorkflow` in `lib/exampleWorkflow.server.ts` — inserted with `active=false`; seed failures are logged and swallowed, never break signup; it counts against the tier workflow cap and is user-deletable like any workflow).
- better-auth owns its DB schema (tables like `user`, `session`, `subscription`) via the shared `pg` Pool in `lib/db.ts`. App-owned tables live in `db/setup.sql` — never redefine better-auth tables there.

Three activation tiers, two billing mechanisms:

- `pro` ($19/mo) and `max` ($79/mo) are Stripe subscriptions managed by the `@better-auth/stripe` plugin (price IDs from `STRIPE_PRICE_PRO`/`STRIPE_PRICE_MAX`).
- Tiers sell **platform limits and built-in model credits**: `PLAN_LIMITS` in `lib/subscription.ts` (workflows 3/20/100, MCP servers 3/10/50, `cronFloorMinutes` 60/5/1 — the tightest schedule a tier may run, `modelCredits` 1,000/15,000/60,000 — per Stripe billing period on paid tiers, rolling 30-day window on free; see "Model access"), enforced server-side in `createWorkflow` and `saveMcpServer` (`limitsFor` treats a not-activated user as `free`). Tier card copy in `app/(saturn)/activate/tierCard.tsx` must stay in sync with `PLAN_LIMITS`. The cron floor (schedules now live in a `schedule` event node, not the workflow row) is enforced two ways: the designer's cron picker caps its offered intervals to `cronFloorMinutes` (`CronBuilder`'s `floorMinutes` prop, fed from the designer page's `limitsFor(level)`), and — the authority — the runner's claim guard silently clamps a downgraded/too-tight cron to the floor at run time (`max(50, floor*60-30)`). There is no save-time hard check anymore (`assertCronFloor` was removed); `validateGraphStrict` only warns on a blank/invalid schedule cron.
- `free` is a non-Stripe tier stored as a `plan` column on `user` (declared as a better-auth `additionalFields` entry with `input: false` so clients can't set it; written by raw SQL in `app/(saturn)/activate/actions.ts` and `app/dashboard/(shell)/upgrade/actions.ts`).
- `lib/subscription.ts` is the single source of tier logic — don't re-derive it:
  - `getActivation(headers)` → effective level: active/trialing Stripe subscription wins (max > pro), then `free`, then `null` (not activated).
  - `getActivationDetails(headers)` adds `status`, `pendingCancel` (checks both `cancelAtPeriodEnd` and `cancelAt` — Stripe portal cancellation only sets the latter), and `periodEnd`.
  - `requireUser()` — session check + redirect to `/onboard`; used by every server action.
  - `getActivationLevels(userIds)` — headless batch resolver for the cron runner (same semantics as `getActivation`, but one direct query against `"user"` + better-auth's `subscription` table instead of request headers).
  - Also exports `LEVEL_RANK`, `isPaidPlan`, `baseUrl`, `PLAN_LIMITS`, `limitsFor`.

Client-side auth uses `authClient` from `lib/auth-client.ts` (`better-auth/react`).

### User registry (MCP servers + skills)

Users register MCP servers and skills at `/dashboard/settings`; entries become designer nodes:

- `registry_entry` table in `db/setup.sql` (uuid id, `user_id` FK, `kind in ('mcp','skill')`, name; emoji/description for skills; `server_url`/`auth_token`/`tools` jsonb for MCPs). `tools` is a per-tool allowlist: `[{name, access: "read"|"write", enabled, readOnly?, description?, params?}]` — access is the user's grant, set per tool via a 3-position switch (off / read / read+write) in the edit modal. `readOnly`/`description`/`params` are discovery-derived and server-side only: `parseTools` strips them from client submissions, `saveMcpServer` re-attaches the stored ones by tool name, and the tool editor serializes only the `{name, access, enabled}` triple. `readOnly` is tri-state: `true`/`false` when the server annotates the tool, absent when it sends no annotations (capability unknown — most servers). `canCallTool` in `lib/registry.ts` is the call-time gate and blocks only a provable mismatch: `readOnly: false` (server explicitly declares write capability) with a read grant. Unknown/manual tools trust the user's grant. The settings 3-position switch disables contradicted positions ("read" on declared-write tools, "read+write" on declared-read-only tools).
- `lib/registry.ts` is the client-safe single source (types, caps `MAX_ENTRIES_PER_KIND`/`MAX_MCP_TOOLS`, `buildUserCatalog` — rows → `CatalogEntry` nodes). Skills key as `skill:<uuid>`; MCP servers expand to **one node per enabled tool** keyed `mcp:<uuid>:<toolName>` (grouped under the server name in the toolbox), plus — when the server has **2+ enabled tools** — a **general-server chip** keyed `mcp:<uuid>:*` (label "All tools", sorted first in the group; `toServerAllToolsEntry`). Both are non-executable **grant chips**: no flow or param ports, just a single value output (`tool` / `skill`) that connects only to an agent node's matching `tools`/`skills` port — MCP tools run exclusively through agents, which build the arguments themselves (the per-param port/config machinery is gone; tool `params` still live on the registry rows and feed `buildToolDefs` server-side). The general-server chip shares the per-tool chip's shape/connection rules (it carries `toolName: "*"` = the `ALL_TOOLS` sentinel from `lib/agent.ts`, so `isMcpChipEntry`/`chipKind` treat it as a tool chip); `executeAgentTurn` expands the `*` ref to the server's every enabled + `canCallTool`-passing tool, **silently skipping** off/write-mismatched ones (never an error, unlike a single-tool grant), deduping against any individually-wired chips of the same server and capping the total at `MAX_GRANTED_TOOLS`. `buildUserCatalog` skips a real tool literally named `*` so it can't collide with the sentinel key. A hidden `legacy: true` generic `mcp:<uuid>` entry is still emitted so graphs saved before per-tool nodes keep resolving (inert at runtime, like every standalone mcp/skill node). The DB query `getUserRegistry` lives in `lib/registry.server.ts` (its SELECT never includes `auth_token` — it projects `has_token`; keep it that way).
- Settings CRUD in `app/dashboard/(shell)/settings/` (`actions.ts`: `saveMcpServer`, `saveSkill`, `deleteRegistryEntry`, `discoverMcpTools`; add/edit modals + two-step delete button; mutations refresh via `revalidatePath("/dashboard/settings")`). `saveMcpServer`/`saveSkill` return `{ error: string }` for expected validation/cap failures (rendered as inline red text by their modals, which keep text fields controlled so a failed submit doesn't wipe input) and return `undefined` on success — never throw for user errors (a thrown server-action error hits Next's generic error page and prod redacts the message). Server URLs must be https and public (`assertHttpsUrlShape` in `lib/mcp.ts` — sync, no DNS — rejects literal private/loopback/localhost hosts at save; see the SSRF guard below); tokens are write-only (blank keeps, checkbox clears). MCP logos come from the URL's **apex** domain via Google s2 favicons (`app/dashboard/mcpLogo.tsx`, letter-tile fallback); skills use the shared emoji picker (`app/dashboard/emojiGrid.tsx`).
- MCP connectivity lives in `lib/mcp.ts` (server-only): Streamable-HTTP `tools/list` discovery and the MCP OAuth 2.1 flow (protected-resource metadata → AS metadata → dynamic client registration → PKCE). **SSRF guard:** the `server_url` and every metadata-derived endpoint (authorization_servers, registration/token URLs from server-returned JSON) are attacker-influenceable, so every outbound `fetch` (`rpc`, `fetchJson`, `registerClient`, `tokenRequest`) calls `assertPublicHttpsUrl` first — https-only + DNS-resolve-and-reject any private/loopback/link-local/reserved address (blocks cloud-metadata/RFC1918/localhost; DNS rebinding is blunted, not eliminated). Never add a fetch here that skips it. `discoverMcpTools` pulls the tool list (manual bearer token wins, else stored OAuth token with refresh); a 401 redirects the browser to the authorization server, and `app/api/mcp/oauth/callback/route.ts` (session + state-nonce checked) exchanges the code, stores tokens in the `oauth` jsonb column (server-only, like `auth_token`), and re-runs discovery. Some servers (Google's Gmail MCP) answer discovery anonymously and only 401 at `tools/call`, so a tokenless discovery success also probes protected-resource metadata (`probeAuthServerMeta` — PRM-only, no origin-AS fallback) and starts the same OAuth flow when found; no PRM means public server. The flow is spec-only: an AS without RFC 7591 dynamic client registration can't connect via OAuth (no per-provider pre-registered clients) — use a manual bearer token for such servers. Connect/discovery failures are non-fatal: `discoverMcpTools` redirects back to `/dashboard/settings?entry=<id>&mcp_error=<message>` and the page renders the message inline on that server's card. Discovered tools replace the allowlist via `mergeTools` (user `enabled`/`access` choices survive, but `readOnlyHint` tools are capped at `read`; new read-only tools start on, new write-capable tools start off; freshly discovered `readOnly`/`description`/`params` always overwrite stored ones). Discovery derives `params` from each tool's `inputSchema` via `deriveParams` in `lib/mcp.ts` — top-level properties only, required-first, capped at 12.
- Designer: `[id]/page.tsx` fetches the user catalog and `designer.tsx` merges it with the static catalog into a `byKey` map threaded through canvas/edges/node. Any node type that doesn't resolve (deleted registry entry, removed static entry) renders as a dimmed "(deleted)" port-less placeholder (`missingEntry`) — graphs referencing them still save (`isWorkflowGraph` accepts any string type ≤ `MAX_NODE_TYPE_LENGTH`; unknown types are inert).

### Model access

Two funding paths for agent-node LLM calls, both through OpenRouter, selected per turn in `executeAgentTurn` (`lib/runner.server.ts` — the single chokepoint for the designer's `callAgentModel` action, cron runs, and the MCP server's `run_workflow`):

- **Built-in model credits** (every activated tier, incl. free): while the tier allowance holds, calls run on the platform key `PLATFORM_OPENROUTER_KEY` (optional env — unset degrades everyone to BYOK). Metering lives in `lib/credits.server.ts` (server-only): the `model_usage` ledger in `db/setup.sql` is the source of truth (no balance column, no reset job); `getCreditUsage(userId)` resolves tier + Stripe billing period headlessly (mirrors `getActivationLevels`; a paid sub with null `periodStart` falls back to a 30-day lookback, never unlimited) and sums `credits` since `periodStart`; `recordUsage` debits `ceil(usage.cost * 1000)` credits per platform-billed turn (1,000 credits = $1; raw cost kept as `cost_microdollars`) and never throws — a failed insert must not fail a completed call. `chatComplete` (`lib/agent.server.ts`) requests `usage: {include: true}` and returns a server-only `usage` field (never folded into the client-crossing `AgentModelResult`). Allowances are `PLAN_LIMITS.modelCredits` (1,000/15,000/60,000). Free tier has no subscription period, so its window is the same 30-day lookback — a rolling month, not a fixed reset. Not-activated users (level `null`) get **no** allowance: `getCreditUsage` zeroes it instead of using `limitsFor`'s null→free mapping (sybil guard — credits require at least picking the free plan on `/activate`). Check-then-call-then-record is deliberately non-transactional — concurrent turns can overshoot by ~one 4096-max_tokens turn each; accepted. The used/allowance bar is the shared `CreditsBar` (`app/dashboard/creditsBar.tsx`), rendered whenever the allowance is > 0 ("resets {date}" for paid, "past 30 days" for free) on both settings and the Overview usage panel.
- **BYO OpenRouter key** (not-activated users / allowance exhausted): the permanent fallback. Entered in the "Models" section of `/dashboard/settings`, stored in the app-owned `user_secret` table (`db/setup.sql`, one row per user, `openrouter_key` column). Write-only like `auth_token`: blank keeps, checkbox clears; the client only ever sees a boolean. `saveOpenrouterKey` in `app/dashboard/(shell)/settings/actions.ts`; server-side reads via `getOpenrouterKey`/`hasOpenrouterKey` in `lib/openrouter.server.ts`. Neither key ever leaves the server; BYOK turns are never recorded in the ledger. No key and no credits → the turn errors as a value (upgrade/add-key hint).

When the user has a key set **or** is on an activated tier with a credit allowance (`limitsFor(level).modelCredits > 0` — so any activated tier today, but never level `null`), the designer toolbox lists OpenRouter's models (`listOpenrouterModels` in `lib/openrouter.server.ts` — public `GET https://openrouter.ai/api/v1/models`, deliberately unauthenticated so the 1h-revalidate Next data cache is shareable) as drag-spawnable model nodes. Each `OpenrouterModel` also carries `outputModalities` (`architecture.output_modalities` filtered to `text`/`image`), which drives the agent node's dynamic `output` select, and `supportsReasoning` (`supported_parameters` includes `"reasoning"`), which drives the agent node's dynamic `reasoning` select.

### Workflows

Users create event-driven workflows edited in a node-graph designer (new users start with the seeded inactive example workflow — see the auth section). Workflows trigger from **event nodes** (catalog category `events`) — the entry point is the `schedule` ("scheduled to run") node whose `config.cron` holds a 5-field cron; entry resolution is catalog-driven (`byKey[type].category === "events"`), so future events (a Discord "mentioned" event) are one CATALOG entry + one trigger with no interpreter/designer change. A workflow holds **at most one event node** — the designer disables the event chip / drops event copies once one exists, and `validateGraphStrict` errors on two+ (the interpreter still walks every event-category node, so it stays correct if a legacy multi-event graph loads). `active` is the master switch for whether any events fire (it gates scheduled runs; manual/test runs ignore it):

- `workflow` table in `db/setup.sql` (uuid id, `user_id` FK, name/emoji/description, `cron` — **vestigial/nullable**, the schedule now lives in a `schedule` node's `config.cron` in the graph, `graph` jsonb, `active` boolean — the master event switch: gates **scheduled** (and future live) event runs only, default true, `last_run_at` — the runner's claim timestamp). Run history lives in `workflow_run` (status running/success/error, error text, `log` jsonb of `ConsoleLine`s, trigger cron/manual; no `user_id` — deletion cascades user→workflow→runs and every query reaches runs through an ownership-checked workflow). All workflow queries scope by `user_id`.
- `/dashboard/workflows` — list + `workflowModal.tsx`, one modal for both create and edit of **metadata only** (name/emoji/description — no cron; the schedule is authored in the graph) via `createWorkflow`/`updateWorkflow` in `app/dashboard/(shell)/workflows/actions.ts` (new workflows insert an empty graph `{"nodes":[],"edges":[]}`; expected failures — invalid fields, workflow cap — come back as an `{ error: string }` return value that the modal renders inline and success is `undefined` + redirect/revalidate, never a throw). `cronBuilder.tsx` (moved out of the modal) is a callback component (`onChange(cron)` + `floorMinutes` cap) hosted by the designer's cron popover; it emits a 5-field cron (frequencies minutes/hourly/daily/weekly/monthly — "minutes" emits `* * * * *` or `*/n * * * *` for n ∈ 5/10/15/30; UTC). `lib/cron.ts` grammar: `*` or a plain int per field, plus `*/n` (2–30) in the **minute field only**; `isValidCron` validates, `describeCron` humanizes the builder's shapes (incl. "every minute"/"every N minutes"; shown under the schedule node and in the topbar's test-event dropdown), `cronMatches` (UTC fields, plain AND) drives the runner, `cronMinIntervalMinutes`/`nextCronOccurrence` remain exported but are no longer wired into save/overview. Each list card shows a last-run status chip (green/red/pulsing-gray dot + relative time, "never run") linking to the runs page, plus an active pill switch (`activeToggle.tsx`, optimistic, calls `setWorkflowActive` — explicit desired state) that pauses/resumes scheduled execution (the old per-card cron chip is gone). Delete via `deleteWorkflow` (idempotent), surfaced by the shared two-step `app/dashboard/deleteWorkflowButton.tsx` on list cards and the designer topbar.
- Scheduled execution: `lib/runner.server.ts` (server-only). `runDueWorkflows` — the per-minute tick behind `/api/cron` — sweeps orphaned `running` rows (janitor), selects candidates (`active` workflows whose graph contains a `schedule` node via jsonb containment `graph->'nodes' @> '[{"type":"schedule"}]'`, loading the graph so the cron can be read from each schedule node's `config.cron`; the claim UPDATE re-checks `active`), JS-filters via `matchingScheduleNodeIds(graph, now)` = schedule nodes whose cron `cronMatches` the current UTC minute (**no missed-minute catch-up**), batch-resolves tiers via `getActivationLevels`, then **claims** each candidate with a single-statement conditional UPDATE on `last_run_at` (atomic on pgbouncer transaction pooling; the guard `max(50, floor*60-30)` doubles as the run-time tier-floor clamp). Claimed runs execute via the shared core `executeWorkflowRun(wf, {trigger, timeoutMs?, entryNodeIds?})` — passing the matching schedule node ids as `entryNodeIds` so the run fires only the schedule(s) due this minute; it records the `workflow_run` (trigger `cron` for the tick, `manual` for the MCP server's `run_workflow` — which passes no `entryNodeIds`, firing all event nodes), builds the byKey catalog, captures the capped log (300 lines × 2000 chars), runs the interpreter under a 240s `AbortController`, persists status/error/log, prunes to the newest 50 per workflow, and never rejects. Tick runs are capped at 25/tick. Also exports the userId-parameterized execution cores `executeMcpTool`/`executeAgentTurn` (semantic checks + grant re-resolution live here — graph/agent input is untrusted server-side too); the designer's `callMcpTool`/`callAgentModel` actions are thin wrappers (`requireUser()` + client-payload shape validation → delegate). Runs page at `/dashboard/workflows/[id]/runs` (inside `(shell)` — session + ownership checked in the page, `<details>` log viewer); the designer topbar links to it next to the test-event runner.
- `/dashboard/workflows/[id]` — designer: `designer.tsx` (reducer + undo/redo in `graphReducer.ts`, dirty tracking, debounced autosave flushed on unmount, Cmd/Ctrl+S save, Cmd/Ctrl+A select-all, Cmd/Ctrl+D duplicate, arrow-key nudge, Escape cancels drags), `canvas.tsx` (pan/zoom/marquee, zoom-to-fit on load + ⛶ button), `node.tsx`, `edges.tsx`, `toolbox.tsx` (searchable). Nodes snap to the 24px `GRID` on spawn and drag-end. `node.tsx` and the per-edge path are `React.memo`ized — node gesture handlers read the live graph/selection through refs (`graphRef`/`selectionRef`), not props; keep new per-render props off `Node` or the memo dies. Config fields with `overriddenBy` dim when their port is connected. The **topbar test-event runner** (`topbar.tsx`) replaces the old ▶ run button: a `<select>` of the graph's event nodes (labelled by `describeCron`) + a run button; the designer runs `runWorkflow` with `{ entryNodeIds: [selectedEventId] }`. Clicking a `schedule` node opens the **cron popover** (`[id]/cronPopover.tsx`, hosting `CronBuilder` in callback mode; the same fixed-position + backdrop shell as `pathPicker.tsx`, gated by `cronFloorMinutes` passed from the page) which writes `config.cron` as one undo step; the node's label strip shows `describeCron(config.cron)`. Saves via `saveWorkflow` server action (validates UUID, `isWorkflowGraph` — unique node ids, edge endpoints exist — plus node/edge count caps and a serialized-size cap; the one-event-per-workflow rule is a designer/`validateGraphStrict` semantic check, not part of this shape guard).
- `lib/workflow.ts` — node catalog (`CATALOG` holds the built-in nodes split across the `events` (`schedule` — the scheduled entry point carrying `config.cron`; plus a hidden `legacy: true` `start` so pre-events-framework graphs still resolve), `logic` (`if`/`loop`/`and`/`or`/`not`/`await`), and `data` (`string`/`number`/`extract`/`print`) categories — each with its own color in `CATEGORY_STYLES` — plus the static `saturn` and `model` categories; mcp/skill nodes come exclusively from the user registry). `string`/`number` are bare header-less **value boxes** (a third non-rectangular shape): the value is edited inline, the box grows with its content (string only; number is a fixed compact box), and a single value output sits on the right edge (`isLiteralEntry`/`nodeWidth`/`nodeHeight` in `geometry.ts`, `node.tsx`'s literal branch — the whole box drags and a press that never becomes a drag focuses the field). They replaced the old single `literal` node (config `valueType` picking string/number), which is kept as a hidden `legacy: true` entry so saved graphs still resolve/run (interpreter keeps its `case "literal"`); the toolbox generic section filters `legacy`. `events` nodes render **non-rectangular** like model nodes: a circle (`EVENT_W`/`EVENT_H`/`EVENT_LABEL_H`/`isEventEntry` in `geometry.ts`; `node.tsx`'s event branch — `entry.emoji` icon centered (schedule = ⏰), single output on the right-edge midpoint), mirroring the model circle branch. When the entry has a `cron` config field (schedule), the label strip shows `describeCron(config.cron)` ("not scheduled" when blank) and a press-not-drag click fires `onOpenCron` → the designer's cron popover. Also holds graph types, and connection rules (`canConnect`, `edgesToReplace` — both take an optional combined-catalog map so user registry nodes connect too). Flow outputs may fan out (multiple edges — the interpreter runs the branches concurrently); value inputs stay single-edge except ports marked `PortSpec.multi` (`await.values` and the agent's `tools`/`skills`); ports marked `PortSpec.accepts` (`"tool" | "skill"` — the agent's grant inputs) take only outputs of matching mcp/skill chip nodes, and chip outputs connect nowhere else (`canConnect` gates both directions). Also here: the shared persistence caps `MAX_NODES`/`MAX_EDGES`/`MAX_GRAPH_JSON` (used by `saveWorkflow` and the MCP `save_graph`/`validate_graph`) and `validateGraphStrict(graph, byKey)` — the deep validator for graphs authored without the designer's UI guardrails (hard errors for bad ports/kind mismatches/duplicate edges/value-input fan-in/non-chip sources into `accepts` ports/more than one event node; warnings for unknown types, no event node, blank/invalid schedule cron, chip outputs into ordinary value ports, over-cap grants).
- `geometry.ts` is the single source of truth for node metrics; `node.tsx` and `edges.tsx` must match it exactly — edge anchors are computed, never DOM-measured. Config rows are per-field height (`configRowHeight`: `CONFIG_ROW_H` 36, textarea fields `TEXTAREA_ROW_H` 72 — the `h-[72px]` row in `node.tsx`). A node box must never carry its frame as a real CSS `border`: ports anchor on the border box (`node.x`/`node.y`) while absolutely-positioned children anchor on the padding box, so a border pushes every marker inward by its width and grows the box past `nodeHeight()` — paint frames with `nodeFrame.tsx`'s inset overlay instead, and hang perimeter ports off the borderless outer box (see the model/event/integration branches). Port markers: flow ports are a filled diamond (45°-tilted square), value ports a hollow `○`; both are rotation-agnostic, so no branch rotates them.
- Saturn node (`agent`, cyan, toolbox section "agents"): an LLM agent loop, fully **port-driven** — value inputs `prompt`, `system`, `model` (usually fed by string/model nodes) plus the multi grant inputs `tools` and `skills` (`accepts: "tool" | "skill"` — fed by mcp/skill chip outputs). Grants resolve **statically from the source node's type** (`toolRefFromNodeType`/`skillIdFromNodeType` in the client-safe `lib/agent.ts`, which also hosts `MAX_GRANTED_TOOLS` 20 / `MAX_GRANTED_SKILLS` 10 — the interpreter dedups and warn-slices at the caps); chips are never evaluated as values. Two config fields, both `dynamicOptions` selects gated on the resolved model (connected model node wins over the literal, same as the interpreter; unknown/empty slug or non-model upstream → empty disabled select; canvas resolves the slug once via `resolveAgentModelSlug` and computes both option strings, mirroring the `overriddenIds` comma-string-prop pattern so `Node`'s memo survives — `node.tsx` picks the right string by `field.id`): `output` (options = the model's `outputModalities`) and `reasoning` (options gated on `supportsReasoning` — capable model → `off/low/medium/high`, known non-reasoning model → `off` only, unknown slug → locked). The `reasoning` mode threads as a raw string through `CallAgentRequest`; `executeAgentTurn` allowlists it against `{off,low,medium,high}` and maps it to OpenRouter's `reasoning` param (`off` → `{enabled:false}`, effort levels → `{effort}`, blank/invalid → omitted = model default), dropping it entirely for `output=image` (single-turn); non-reasoning/unknown models with a saved mode pass through and let OpenRouter handle it, like `output=image` on a text model. Legacy: `config.system`/`config.model` from old graphs are still honored as interpreter fallbacks when the matching port has no edge, but old JSON config grants (`config.tools`/`config.skills`) no longer apply — the grant picker and its parsers are gone; `output=image` sends `modalities: ["image","text"]`, ignores tool grants with a console warning (image models don't accept `tools`, so the loop is single-turn), and puts the first returned `data:image/` URL on `result` (capped at `MAX_IMAGE_DATA_URL` 4 MB in `executeAgentTurn`, riding its own `image` field past the `MAX_MODEL_CONTENT` 20k slice — never fold it into `content`); no image returned → warn + fall back to text. Any other `output` value (including legacy `"plan"` — the plan pipeline was removed with the swarm node) runs as text; `result` carries the raw final text. (Old graphs containing a `swarm` node render it as an inert "(deleted)" placeholder; edges into a removed port don't render and are inert.) Model node (`model`, rose, toolbox section "models", always the last section): a pure-value node emitting `config.model` on its `model` output — one static node type; the toolbox's per-OpenRouter-model chips just prefill `config.model` on spawn (a `SpawnPreset` threaded through `toolbox.tsx` → `designer.tsx`), so graphs never reference per-model keys. Model nodes render **circular** (one of four non-rectangular node shapes, the others being event circles, literal value boxes, and mcp/skill grant chips — see the events/data/registry sections; chips are rounded squares, `MCP_CHIP` 60px with the server favicon / `SKILL_CHIP` 48px with the emoji, category-color `border-2`, an `CHIP_LABEL_H` label strip below, single output on the right-edge midpoint — `isMcpChipEntry`/`isSkillChipEntry`/`chipSize` in `geometry.ts`, `node.tsx`'s chip branch): the toolbox lists models as a 3-column grid of 48px logo circles (`ModelChip`), and on canvas the node is a 72px circle (`MODEL_D`/`MODEL_LABEL_H`/`isModelEntry`/`nodeWidth` in `geometry.ts`; the output anchors at the circle's right-edge midpoint) with the company logo inside (`[id]/modelLogo.tsx` — slug's author segment → apex-domain map → Google s2 favicon, robot-icon fallback for unknown authors/failed loads) and the name underneath. Per-model chips spawn with `config.preset = "1"` → read-only name; the blank "model" chip (and legacy graphs without the flag) keep an editable slug input under the circle. Loaded skills = the registry skill's `description` text injected into the system prompt **server-side by id** (client-sent instruction text is never trusted). Execution lives in the interpreter (`runAgentLoop`): one `callAgent` hook invocation per LLM turn — `executeAgentTurn` in `lib/runner.server.ts` (validates model id/system size, re-resolves every tool grant against the registry with `enabled` + `canCallTool` — rejects on any mismatch, appends skill prompts, calls OpenRouter via `lib/agent.server.ts` `chatComplete`; wire-format tool names are built and decoded server-side by `buildToolDefs`), reached in the designer through the `callAgentModel` action (which adds transcript shape+size validation of the browser-built payload) — and each model tool call executes through the `callMcp` hook (`callMcpTool` action / `executeMcpTool` core). Caps: `MAX_AGENT_TURNS` 8, `MAX_TOOL_CALLS_PER_TURN` 5, interpreter-side `MAX_AGENT_MCP_CALLS` 40, `MAX_MODEL_RESULT_CHARS` 8000 fed back to the model.
- Integration nodes (orange, toolbox group "Apps", keys `integration:<provider>`): outbound message nodes (flow in + `message` value port overriding the `message` config literal, flow out) generated from the provider descriptors in client-safe `lib/integrations.ts` — first provider `discord-webhook`, webhook URL pasted into `config.webhookUrl` (stored plaintext in graph jsonb; the user's own graph, accepted). Sends execute server-side in `lib/integrations.server.ts` `executeIntegration(userId, providerId, config, message)` — errors as values; the per-provider exact-host allowlist is the SSRF guard (Discord: https + `discord.com`/`discordapp.com` host + `/api/webhooks/` path prefix, never loosen to substring checks; content truncated to Discord's 2000-char cap). A `message` that is a `data:image/…;base64,…` data URL (e.g. an agent node's `output=image` result wired into the `message` port) is uploaded as a **multipart `files[0]` attachment** instead of text (decoded to bytes, filename ext from the mime subtype, `DISCORD_UPLOAD_LIMIT` 8 MiB backstop); `executeIntegration`'s length gate exempts image data URLs from the 4096-char `MAX_INTEGRATION_MESSAGE` cap, allowing up to `MAX_INTEGRATION_IMAGE` (4 MiB, mirrors the runner's `MAX_IMAGE_DATA_URL`) — through the interpreter's `callIntegration` hook, wired at the usual three sites (`callIntegration` action in `[id]/actions.ts` for test runs, `executeWorkflowRun` for cron/manual). Budget `MAX_INTEGRATION_CALLS` 20. `validateGraphStrict` warns on blank `requiredConfig` fields. Integrations all carry the `integration` category, but each descriptor carries two more required fields that split **grouping** from **color**: `app` names the app (`discord`) and heads the node's toolbox subsection in the Apps group (grouped by first appearance in `INTEGRATIONS`; a provider's `label` is then the bare node name — `send webhook`, not `discord webhook`), while `section` names one of `INTEGRATION_SECTIONS` in `lib/integrations.ts` (`events`/`logic`/`data` — a subset of `NodeCategory`, enforced by `satisfies`; `discord-webhook` is `data`) and supplies only the **color**, mirroring the Blocks group — a discord webhook paints teal like the `print` node. The two ride along as `CatalogEntry.group` (which the MCP `get_catalog` tool reports as `app` for integrations, `server` for mcp entries) and `CatalogEntry.section`. Resolve colors through **`entryStyles(entry)`** (`lib/workflow.ts` → `CATEGORY_STYLES[entry.section ?? entry.category]`), never by indexing `CATEGORY_STYLES` with `entry.category`, or integrations revert to orange; `CATEGORY_STYLES.integration` (orange) now only backs the "(deleted)" placeholder from `missingEntry`, which has no section. `edges.tsx` resolves the stroke once when building an `End` (`color`, not a category) since a node's category no longer determines its color. Adding a provider (Telegram, Discord bot, Slack…) = one descriptor in `lib/integrations.ts` + one sender in `lib/integrations.server.ts`; interpreter/designer/toolbox need no changes (a provider whose color category isn't in `INTEGRATION_SECTIONS` yet also needs its name added there; a new `app` value just heads a new subsection). `lib/integrations.ts` must only `import type` from `lib/workflow.ts` (workflow.ts value-imports it — a value import back would cycle at runtime).
- Test run: the topbar's test-event runner (event `<select>` + ▶ run) executes the in-memory graph client-side via `lib/interpreter.ts` `runWorkflow(graph, byKey, hooks, { entryNodeIds? })` — entries are the given ids (the selected event), else all event-category nodes; each runs its `"out"` flow concurrently (`Promise.allSettled`). Shared with the server cron runner (pure graph-walking logic, all side effects injected as hooks `emit`/`callMcp`/`callIntegration`/`callAgent`/`onValue`/`signal`; exact flow-cycle detection per chain, step- and MCP-call-capped, per-step-local value memoization + cycle stack so concurrent branches can't clobber each other, stoppable — the button becomes ■ stop, wired through an `AbortSignal`) and streams output into the `[id]/console.tsx` bottom panel (drag-resizable via its top edge — height state lives in `designer.tsx`, in-memory only, never persisted; sticks to the bottom only while already there; `ConsoleLine` kind `"image"` — `text` is a data URL, emitted by the print node for `data:image/` values — renders as an inline `<img>`; the cron runner's `emit` persists it as a `describeImage` info placeholder `[image · mime · KB]` instead, never base64 into `workflow_run.log`). Flow fan-outs run branches concurrently (`Promise.allSettled`; the first failure aborts the run and siblings stop at their next step). The `await` node is the join barrier: the last arriving incoming flow edge continues the chain and fills `results` with a JSON array of its `values` edges (edge order); a barrier an upstream `if` diverged past warns "await never completed (k/n branches)" once the outermost fan-out settles. Standalone mcp and skill nodes are not executable — the interpreter skips them with a warning (grant chips only feed agent ports; old flow-wired MCP graphs degrade to that skip). Agent tool calls execute for real through the `callMcpTool` server action in `[id]/actions.ts` (allowlist- and `canCallTool`-checked, tokens stay server-side via `freshMcpToken` in `lib/registry.server.ts`, tool call in `lib/mcp.ts` `callTool`). Scheduled runs execute the same interpreter server-side via `lib/runner.server.ts` (see above). Test runs also report every computed value through the interpreter's `onValue` hook into a designer-held samples map (in-memory only, never persisted); the extract node's `path` field (`picker: "json-path"` on its `ConfigField`) gets a `{}` button that opens `[id]/pathPicker.tsx` — a JSON tree of the upstream node's last output where clicking a leaf writes the dot-path as one undo step.

### Hosted MCP server (`/mcp`)

External agents (e.g. Claude Code: `claude mcp add --transport http saturn <baseUrl>/mcp`; the shared "Connect an agent" card (`app/dashboard/connectAgent.tsx`, rendered on Overview and settings) lists commands for Claude Code, Codex, Gemini CLI, VS Code, and Cursor) edit and test-run the authenticated user's workflows:

- `app/mcp/route.ts` — stateless Streamable-HTTP JSON-RPC handler (`maxDuration` 300): hand-rolled like the client in `lib/mcp.ts`, plain JSON responses (no SSE, no session id, no batching), methods `initialize` (version negotiation across 2024-11-05…2025-06-18) / `ping` / `tools/list` / `tools/call`; notifications → 202; `GET`/`DELETE` → 405. `POST` is wrapped in `withMcpAuth(auth, …)` — the bearer token resolves to an `OAuthAccessToken` whose `userId` is the only identity input; a missing/invalid token 401s with the `WWW-Authenticate` resource-metadata pointer.
- OAuth discovery: the root `.well-known` routes (see Auth section) serve the AS + protected-resource metadata; the actual authorize/token/register endpoints live under the better-auth catch-all.
- `app/mcp/protocol.ts` — JSON-RPC envelopes/negotiation; `app/mcp/tools.ts` — 11 tool definitions (hand-written JSON Schemas) + `dispatchTool(userId, name, args)`: `list_workflows`, `get_workflow`, `create_workflow`/`update_workflow` (metadata only, workflow cap via `getActivationLevels` — no cron param; the schedule is authored in the graph via `save_graph`; update also takes the boolean `active`), `delete_workflow`, `get_catalog` (static catalog + `buildUserCatalog`, minus `legacy`/`missing` entries, plus an embedded graph-authoring guide), `validate_graph`/`save_graph` (`isWorkflowGraph` + size caps + `validateGraphStrict`; save rejects on hard errors, returns warnings), `run_workflow` (→ `executeWorkflowRun` with `trigger='manual'` — the only writer of that trigger value; real side effects, same as a scheduled run), `list_runs`, `get_run` (ownership via workflow join). Tool failures return as `isError` results, never JSON-RPC errors.

### SEO

- `lib/seo.ts` is the single source of site identity (`SITE_NAME`/`SITE_TAGLINE`/`SITE_TITLE`/`SITE_DESCRIPTION`/`GITHUB_URL`/`ORG_NAME`/`siteUrl`). It reads `BETTER_AUTH_URL` directly (fallback `VERCEL_PROJECT_PRODUCTION_URL`, then localhost) — never import `lib/subscription.ts`'s `baseUrl` here or in the metadata routes, it would drag auth/db/Stripe into their bundles. `BETTER_AUTH_URL` doubles as the canonical site URL.
- Root metadata + `viewport` (themeColor light/dark pair matching `globals.css`) live in `app/layout.tsx`; title template is `%s · Saturn`. `openGraph.images`/`twitter.images`/`icons`/`manifest` fields are deliberately absent — the file conventions (`app/opengraph-image.tsx`, `app/twitter-image.tsx` re-exporting it, `app/favicon.ico` — root-only multi-res 16/32/48 ICO, the primary Google-search favicon fetched at `/favicon.ico`; `app/icon.png` 512² + `app/icon1.png` 192² — numbered-suffix multiple `<link rel=icon>`s, 192 is a 48-multiple for Google's size preference; `app/apple-icon.png` 180², `app/manifest.ts`) override the metadata object. The OG image renders with satori via `ImageResponse` from `next/og` (GeistMono-Regular.ttf from node_modules; the landing divider's `≡`/`≣` glyphs are missing from that font and render as tofu — keep them out of OG art).
- Indexing policy: only `/`, `/terms`, and `/privacy` are indexed (canonicals + the only `app/sitemap.ts` entries). `/onboard` and `/activate` are meta-noindexed but stay crawlable (not in robots.txt, so the noindex is visible). All of `/dashboard/**` is noindexed by the metadata-only pass-through `app/dashboard/layout.tsx` **and** disallowed in `app/robots.ts` (along with `/api/`, `/mcp`; `/.well-known` stays crawlable for OAuth discovery).
- Landing JSON-LD (`app/(saturn)/page.tsx`, `@graph` of Organization/WebSite/SoftwareApplication) carries offer prices that must stay in sync with `TIERS` in `tierCard.tsx`.
- No em-dashes in marketing/SEO copy (meta descriptions, OG text, JSON-LD strings).

### Conventions

- Server actions are public POST endpoints — every action re-checks the session itself via `requireUser()` (see `app/(saturn)/activate/actions.ts`).
- Components colocate with the route that uses them (e.g. `app/dashboard/sidebar.tsx`); only cross-cutting code goes in `lib/`.
- Animations respect `prefers-reduced-motion` throughout (scene, moons, transitions, glows).
