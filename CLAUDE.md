## Commands

- `npm run dev` — dev server at http://localhost:3000
- `npm run build` — production build
- `npm run lint` — ESLint
- `psql "$DATABASE_URL" -f db/setup.sql` — create app tables (idempotent; also enables the `vector` extension — pgvector, **required** for the `memory_item` embeddings — and creates `memory_item`; also re-applies the `workflow_run_trigger_check`, `model_usage_source_check`, and `registry_entry` kind widenings — the first two now include `'event'`, the kind CHECK now includes `'memory'` and `'variable'` — via `drop constraint if exists` + `add constraint`, so it must run before deploying code that writes `trigger='event'`/`source='event'` or `kind='memory'`/`kind='variable'`)
- `npx @better-auth/cli@latest migrate --config lib/auth.ts` — create/update better-auth-owned tables (needed once for the mcp plugin's `oauthApplication`/`oauthAccessToken`/`oauthConsent`; `generate` first to review the SQL)
- No test suite exists.

Local Stripe webhooks: `stripe listen --forward-to localhost:3000/api/auth/stripe/webhook` (its signing secret goes in `STRIPE_WEBHOOK_SECRET`). Required env vars are documented in `.env.example`.

Production is self-hosted on a Raspberry Pi (hostname `saturn`, reachable as `saturn.local`) behind a Cloudflare Tunnel serving https://saturn.wilcus.com: `deploy.sh` rsyncs the source to `/srv/saturn/app`, runs `npm ci` + `next build` on the Pi, installs `deploy/saturn.service`, and restarts the `saturn` systemd service (`next start` bound to 127.0.0.1:3000, run as the `saturn` system user, env in `/etc/saturn/saturn.env`). **Background work runs in-process** in production (`instrumentation.ts` → `lib/background.server.ts`, started once per server boot when `NODE_ENV === "production"` — never during builds (Next skips `register()` in the build phase) and never in `next dev`): the per-minute cron scheduler (`lib/scheduler.server.ts` — self-arming :00-aligned tick calling `runDueWorkflows(minute)`, serialized in-process, with missed-minute catch-up capped at 5 so sparse crons recover after stalls; the runner's claim guard collapses catch-up bursts and any overlapping tick source), the Discord Gateway listener (`lib/gateway.server.ts`), and the Telegram long-poller (`lib/telegram.server.ts`) — see "Extension event nodes". There are no external triggers anymore (`deploy.sh` retires a leftover `saturn-events` unit on deploy). In-process caches (`lib/cache.server.ts` + better-auth's session cookieCache) assume this **single-process deployment** — remove/externalize them before ever running more than one instance.

## Stack

Next.js 16 App Router (with `experimental.viewTransition`), React 19, Tailwind CSS 4 (no config file — theme, view-transition keyframes, and tier glow utilities live in `app/globals.css`), TypeScript, better-auth + `@better-auth/stripe`, raw `pg` Pool against Neon Postgres (no ORM). Fonts come from the `geist` package + Google Inter, loaded in `app/layout.tsx`.

`next.config.ts` sets global security headers via `headers()` (applied to `/:path*`): a CSP (`default-src 'self'`; `script-src`/`style-src` allow `'unsafe-inline'` for Next's inline bootstrap + the designer's inline styles; `img-src` allows `data:` + Google s2 favicons + the avatar host; dev additionally allows `'unsafe-eval'` + `ws:` for Fast Refresh/HMR — computed inside `headers()` so `NODE_ENV` is resolved at bake time), plus `frame-ancestors 'none'`/`X-Frame-Options: DENY` (clickjacking, notably the OAuth consent page), HSTS, `nosniff`, `Referrer-Policy`, and a minimal `Permissions-Policy`.

## Architecture

### Auth + billing (the core of the app)

Everything flows through better-auth, configured in `lib/auth.ts`:

- Google OAuth is the only sign-in method.
- `app/api/auth/[...all]/route.ts` is the single catch-all handler for all auth routes **and** the Stripe webhook (`/api/auth/stripe/webhook`) — the stripe plugin handles webhooks; don't add a separate webhook route. The only other route under `app/api/` is the MCP OAuth callback (`app/api/mcp/oauth/callback/route.ts`). Outside `app/api/`: the hosted MCP server at `app/mcp/route.ts` and the root OAuth discovery documents (`app/.well-known/oauth-authorization-server/` + `oauth-protected-resource/` and its `/mcp`-suffixed form) — see "Hosted MCP server" below.
- The `mcp` plugin (better-auth built-in) makes the catch-all an OAuth 2.1 authorization server (dynamic client registration + PKCE at `/api/auth/mcp/authorize|token|register`) for the hosted MCP server; `loginPage: "/onboard"` — an unauthenticated authorize request bounces through Google sign-in and resumes automatically. **Consent is forced** (fixes a cross-account-takeover: the plugin issues a code with no consent unless the request carries `prompt=consent`, which — with anonymous client registration + a Lax session cookie — let a logged-in victim be lured to an authorize URL that silently minted a token for an attacker's client). Root-level **`proxy.ts`** (Next 16's renamed middleware; matcher `/api/auth/mcp/authorize`) rewrites every authorize request to add `prompt=consent`, and `mcp({ oidcConfig: { consentPage: "/oauth/consent" } })` routes it to the first-party consent screen `app/oauth/consent/page.tsx` (server component; Approve/Deny posts through a server action to `auth.api.oAuthConsent`, then redirects to the client's callback). Do not remove the proxy or `consentPage` without replacing the consent gate. The post-login resume path is covered because the plugin replays the already-rewritten query from its signed cookie. Its tables (`oauthApplication`, `oauthAccessToken`, `oauthConsent`) are better-auth-owned — migrate via the CLI, never `db/setup.sql`.
- The `nextCookies` plugin must stay last in the plugins array.
- `session.cookieCache` is enabled (60s maxAge): session data rides a signed cookie, skipping the per-request session SELECT. `user.plan` is in the cached payload, so plan changes/remote sign-outs are stale ≤60s — don't raise the maxAge without revisiting the activate/upgrade flows (`refreshCache` must stay unset; DB-backed setups warn). `getSessionCached` in `lib/subscription.ts` (React `cache()`) additionally dedupes session resolution per request — pages/actions use it instead of calling `auth.api.getSession` directly.
- `databaseHooks.user.create.after` seeds every new user an inactive "Daily haiku" example workflow (`seedExampleWorkflow` in `lib/exampleWorkflow.server.ts` — inserted with `active=false`; seed failures are logged and swallowed, never break signup; it counts against the tier workflow cap and is user-deletable like any workflow).
- better-auth owns its DB schema (tables like `user`, `session`, `subscription`) via the shared `pg` Pool in `lib/db.ts`. App-owned tables live in `db/setup.sql` — never redefine better-auth tables there.

Three activation tiers, two billing mechanisms:

- `pro` ($19/mo) and `max` ($79/mo) are Stripe subscriptions managed by the `@better-auth/stripe` plugin (price IDs from `STRIPE_PRICE_PRO`/`STRIPE_PRICE_MAX`).
- Tiers sell **platform limits and built-in model credits**: `PLAN_LIMITS` in `lib/subscription.ts` (workflows 3/20/100, MCP servers 3/10/50, `memoryStores` 1/5/20, `cronFloorMinutes` 60/5/1 — the tightest schedule a tier may run, `modelCredits` 1,000/15,000/60,000 — per Stripe billing period on paid tiers, rolling 30-day window on free; see "Model access"), enforced server-side in `createWorkflow`, `saveMcpServer`, and `saveMemoryStore` (the memory-store cap; `limitsFor` treats a not-activated user as `free`). Tier card copy in `app/(saturn)/activate/tierCard.tsx` must stay in sync with `PLAN_LIMITS`. The cron floor (schedules now live in a `schedule` event node, not the workflow row) is enforced two ways: the designer's cron picker caps its offered intervals to `cronFloorMinutes` (`CronBuilder`'s `floorMinutes` prop, fed from the designer page's `limitsFor(level)`), and — the authority — the runner's claim guard silently clamps a downgraded/too-tight cron to the floor at run time (`max(50, floor*60-30)`). There is no save-time hard check anymore (`assertCronFloor` was removed); `validateGraphStrict` only warns on a blank/invalid schedule cron.
- `free` is a non-Stripe tier stored as a `plan` column on `user` (declared as a better-auth `additionalFields` entry with `input: false` so clients can't set it; written by raw SQL in `app/(saturn)/activate/actions.ts` and `app/dashboard/(shell)/upgrade/actions.ts`).
- `lib/subscription.ts` is the single source of tier logic — don't re-derive it:
  - `getActivation(headers)` → effective level: active/trialing Stripe subscription wins (max > pro), then `free`, then `null` (not activated).
  - `getActivationDetails(headers)` adds `status`, `pendingCancel` (checks both `cancelAtPeriodEnd` and `cancelAt` — Stripe portal cancellation only sets the latter), and `periodEnd`.
  - `requireUser()` — session check + redirect to `/onboard`; used by every server action.
  - `getSessionCached()` — the request-memoized session resolver (see the cookieCache bullet above).
  - `getActivationLevels(userIds)` — headless batch resolver for the cron runner (same semantics as `getActivation`, but one direct query against `"user"` + better-auth's `subscription` table instead of request headers).
  - Also exports `LEVEL_RANK`, `isPaidPlan`, `baseUrl`, `PLAN_LIMITS`, `limitsFor`.

Client-side auth uses `authClient` from `lib/auth-client.ts` (`better-auth/react`).

### User registry (MCP servers + skills + memory stores + secret variables)

Users register MCP servers and skills at `/dashboard/settings`, persistent agent-memory stores at `/dashboard/memory`, and secret variables in the designer toolbox; entries become designer nodes:

- `registry_entry` table in `db/setup.sql` (uuid id, `user_id` FK, `kind in ('mcp','skill','memory','variable')`, name; emoji/description for skills and memory stores; `server_url`/`auth_token`/`tools` jsonb for MCPs). `tools` is a per-tool allowlist: `[{name, access: "read"|"write", enabled, readOnly?, description?, params?}]` — access is the user's grant, set per tool via a 3-position switch (off / read / read+write) in the edit modal. `readOnly`/`description`/`params` are discovery-derived and server-side only: `parseTools` strips them from client submissions, `saveMcpServer` re-attaches the stored ones by tool name, and the tool editor serializes only the `{name, access, enabled}` triple. `readOnly` is tri-state: `true`/`false` when the server annotates the tool, absent when it sends no annotations (capability unknown — most servers). `canCallTool` in `lib/registry.ts` is the call-time gate and blocks only a provable mismatch: `readOnly: false` (server explicitly declares write capability) with a read grant. Unknown/manual tools trust the user's grant. The settings 3-position switch disables contradicted positions ("read" on declared-write tools, "read+write" on declared-read-only tools).
- **Memory stores** (`kind='memory'`: name/emoji/description, no tools/URL) are persistent agent-memory collections. Their items live in a separate `memory_item` table (`db/setup.sql`, `entry_id` FK, `content`, pgvector `vector(1536)` `embedding`, capped at 2000 items/store — enforced in `memory_save`). Managed on the dedicated Memory tab (`/dashboard/memory` list + `/dashboard/memory/[id]` detail — item browse/search, per-item delete, wipe), **not** settings. `buildUserCatalog` expands each into a fuchsia **grant chip** `memory:<uuid>` (category `memory`, emoji icon, `toMemoryEntry`): no flow ports, a single value output `memory` that connects only to an agent node's single-edge `memory` port — one store per agent. Server core is `lib/memory.server.ts` (see "Model access" / the agent node section). Clicking a **skill or memory** chip on the canvas opens a read-only info popover (emoji/name/description + a one-line wiring hint — `[id]/chipInfoPopover.tsx`, designer-held state via the memo-safe `onOpenInfo` callback; the entry's `description` comes from the registry row through `toSkillEntry`/`toMemoryEntry`); mcp chips open the tool-picker popover instead, so no chip is a dead click. (types, caps `MAX_ENTRIES_PER_KIND`/`MAX_MCP_TOOLS`, `buildUserCatalog` — rows → `CatalogEntry` nodes). Skills key as `skill:<uuid>`; MCP servers expand to **one server node per server** keyed `mcp:<uuid>:*` (`toServerEntry` — label = server name; `toolName: "*"` = the `ALL_TOOLS` sentinel from `lib/agent.ts`, so `isMcpChipEntry`/`chipKind` treat it as a tool chip; always emitted even with 0 enabled tools so saved nodes never flip to "(deleted)"). Both server and skill nodes are non-executable **grant chips**: no flow or param ports, just a single value output (`tool` / `skill`) that connects only to an agent node's matching `tools`/`skills` port — MCP tools run exclusively through agents, which build the arguments themselves (tool `params` live on the registry rows and feed `buildToolDefs` server-side). A server node grants **all enabled + `canCallTool`-passing tools by default**, minus the node's per-node prune in `config.exclude` (a JSON-array-of-strings string, `""`/absent = all; edited via the designer's tool-picker popover — an exclude-list, so tools discovered later are auto-included unless pruned). The entry's `tools` field lists exactly that expansion set (name + description) for the picker and the MCP `get_catalog`. The interpreter attaches the parsed exclusions to the `*` ref (`AgentToolRef.exclude`, validated by `isToolExclusionList`; malformed config warns and grants all) and `executeAgentTurn` expands the ref to the server's enabled + callable tools minus the exclusions, **silently skipping** off/write-mismatched/excluded ones (never an error, unlike a direct per-tool ref, which the API surface still hard-checks), deduping and capping the total at `MAX_GRANTED_TOOLS`. `buildUserCatalog` skips a real tool literally named `*` so it can't collide with the sentinel key. The old per-tool nodes (`mcp:<uuid>:<toolName>`) and the legacy generic `mcp:<uuid>` entry are **gone**: those node types render as inert "(deleted)" placeholders and grant nothing — the interpreter gates grants on the source resolving to a live mcp chip entry, not on the type parse (a legacy generic node that was flow-wired now ends its chain at the skip warning). The DB query `getUserRegistry` lives in `lib/registry.server.ts` (its SELECT never includes `auth_token` — it projects `has_token`; keep it that way) behind a 60s in-process TTL cache (`lib/cache.server.ts`) — every registry mutation must call `invalidateUserRegistry(userId)` (the settings actions and the MCP OAuth callback do); `getMcpSecrets`/`freshMcpToken` stay uncached.
- **Secret variables** (`kind='variable'`: name only, secret in the write-only `auth_token` column — flat `MAX_ENTRIES_PER_KIND` cap, no tier gating) are named user secrets (bot tokens etc.) managed **in the designer toolbox**, not settings: a pinned "variables" split at the bottom of the toolbox aside (visible on every group tab, filtered by the toolbox search) lists them with per-row edit + "+ add", and clicking a `variable:<uuid>` value box on the canvas opens the same `[id]/variableModal.tsx` (skillModal conventions — controlled inputs, inline `{error}`, password value field "blank keeps / checkbox clears", two-step delete; success → `router.refresh()`). The modal is **hosted by `designer.tsx`** (lifted out of the toolbox so both entry points share one instance): the toolbox signals open-intent via an `onEditVariable(row | "new")` prop, and a canvas node click routes through the memo-safe `onOpenVariable(nodeId)` callback, which resolves the row by uuid (`variableIdFromNodeType`) against a `variablesRef` live mirror. Actions `saveVariable`/`deleteVariable` live in `[id]/actions.ts` and call `invalidateUserRegistry`. `buildUserCatalog` expands each into a violet **read-only value box** `variable:<uuid>` (`toVariableEntry` — no inputs, single plain `value` output that connects to any ordinary value input; rendered literal-box-style sized from the *name*, `isVariableEntry` in `geometry.ts`, key-glyph branch in `node.tsx`). **The plaintext never reaches the client**: the interpreter evaluates the node to the opaque sentinel `{{var:<uuid>}}` (`variableSentinel`/`variableIdFromNodeType` in `lib/registry.ts`), so configs, logs, prints, `onValue` samples, and agent prompts only ever carry the sentinel; substitution happens only where secrets are consumed: **`executeIntegration`** (`substituteVariables` in `lib/integrations.server.ts` — regex-scan config values + message, one uncached `getVariableValues(userId, ids)` batch from `lib/registry.server.ts`, userId-scoped so foreign/deleted uuids stay literal and the provider validator rejects them normally; runs before the per-provider SSRF regexes so a substituted token validates) and **`getEventSubscriptions`** (`lib/events.server.ts` — resolves event-node config wired from variable/string nodes at subscription-build time, per workflow owner; see "Extension event nodes"). Cron/manual/event runs are covered via the same hook; everywhere else the sentinel passes through literally (documented in `GRAPH_DOCS`).
- Settings CRUD in `app/dashboard/(shell)/settings/` (`actions.ts`: `saveMcpServer`, `saveSkill`, `deleteRegistryEntry`, `discoverMcpTools`; add/edit modals + two-step delete button; mutations refresh via `revalidatePath("/dashboard/settings")`). `saveMcpServer`/`saveSkill` return `{ error: string }` for expected validation/cap failures (rendered as inline red text by their modals, which keep text fields controlled so a failed submit doesn't wipe input) and return `undefined` on success — never throw for user errors (a thrown server-action error hits Next's generic error page and prod redacts the message). Server URLs must be https and public (`assertHttpsUrlShape` in `lib/mcp.ts` — sync, no DNS — rejects literal private/loopback/localhost hosts at save; see the SSRF guard below); tokens are write-only (blank keeps, checkbox clears). MCP logos come from the URL's **apex** domain via Google s2 favicons (`app/dashboard/mcpLogo.tsx`; on s2 failure a server hosted on the app's own domain falls back to the same-origin `/icon.png`, then the letter tile); skills use the shared emoji picker (`app/dashboard/emojiGrid.tsx`).
- MCP connectivity lives in `lib/mcp.ts` (server-only): Streamable-HTTP `tools/list` discovery and the MCP OAuth 2.1 flow (protected-resource metadata → AS metadata → dynamic client registration → PKCE). **SSRF guard:** the `server_url` and every metadata-derived endpoint (authorization_servers, registration/token URLs from server-returned JSON) are attacker-influenceable, so every outbound `fetch` (`rpc`, `fetchJson`, `registerClient`, `tokenRequest`) calls `assertPublicHttpsUrl` first — https-only + DNS-resolve-and-reject any private/loopback/link-local/reserved address (blocks cloud-metadata/RFC1918/localhost; DNS rebinding is blunted, not eliminated). Never add a fetch here that skips it. `discoverMcpTools` pulls the tool list (manual bearer token wins, else stored OAuth token with refresh); a 401 returns the authorize URL and the settings `connectButton.tsx` navigates there via `window.location` (never a server-action `redirect()` — a same-origin authorize URL, i.e. registering saturn's own MCP server, would make Next's router swallow the redirect chain and strand the consent page on the API URL), and `app/api/mcp/oauth/callback/route.ts` (session + state-nonce checked) exchanges the code, stores tokens in the `oauth` jsonb column (server-only, like `auth_token`), and re-runs discovery. Some servers (Google's Gmail MCP) answer discovery anonymously and only 401 at `tools/call`, so a tokenless discovery success also probes protected-resource metadata (`probeAuthServerMeta` — PRM-only, no origin-AS fallback) and starts the same OAuth flow when found; no PRM means public server. The flow is spec-only: an AS without RFC 7591 dynamic client registration can't connect via OAuth (no per-provider pre-registered clients) — use a manual bearer token for such servers. Connect/discovery failures are non-fatal: `discoverMcpTools` redirects back to `/dashboard/settings?entry=<id>&mcp_error=<message>` and the page renders the message inline on that server's card. Discovered tools replace the allowlist via `mergeTools` (user `enabled`/`access` choices survive, but `readOnlyHint` tools are capped at `read`; new read-only tools start on, new write-capable tools start off; freshly discovered `readOnly`/`description`/`params` always overwrite stored ones). Discovery derives `params` from each tool's `inputSchema` via `deriveParams` in `lib/mcp.ts` — top-level properties only, required-first, capped at 12.
- Designer: `[id]/page.tsx` fetches the user catalog and `designer.tsx` merges it with the static catalog into a `byKey` map threaded through canvas/edges/node. Any node type that doesn't resolve (deleted registry entry, removed static entry) renders as a dimmed "(deleted)" port-less placeholder (`missingEntry`) — graphs referencing them still save (`isWorkflowGraph` accepts any string type ≤ `MAX_NODE_TYPE_LENGTH`; unknown types are inert).

### Model access

Two funding paths for agent-node LLM calls, both through OpenRouter, selected per turn in `executeAgentTurn` (`lib/runner.server.ts` — the single chokepoint for the designer's `callAgentModel` action, cron runs, and the MCP server's `run_workflow`):

- **Built-in model credits** (every activated tier, incl. free): while the tier allowance holds, calls run on the platform key `PLATFORM_OPENROUTER_KEY` (optional env — unset degrades everyone to BYOK). Metering lives in `lib/credits.server.ts` (server-only): the `model_usage` ledger in `db/setup.sql` is the source of truth (no balance column, no reset job); `getCreditUsage(userId)` resolves tier + Stripe billing period headlessly (mirrors `getActivationLevels`; a paid sub with null `periodStart` falls back to a 30-day lookback, never unlimited) and sums `credits` since `periodStart`, behind a 60s in-process TTL cache that `recordUsage` increments in place on success (same-process balance checks stay exact; external writes are stale ≤60s) and drops on insert failure; `recordUsage` debits `ceil(usage.cost * 1000)` credits per platform-billed turn (1,000 credits = $1; raw cost kept as `cost_microdollars`) and never throws — a failed insert must not fail a completed call. `chatComplete` (`lib/agent.server.ts`) requests `usage: {include: true}` and returns a server-only `usage` field (never folded into the client-crossing `AgentModelResult`). Allowances are `PLAN_LIMITS.modelCredits` (1,000/15,000/60,000). Free tier has no subscription period, so its window is the same 30-day lookback — a rolling month, not a fixed reset. Not-activated users (level `null`) get **no** allowance: `getCreditUsage` zeroes it instead of using `limitsFor`'s null→free mapping (sybil guard — credits require at least picking the free plan on `/activate`). Check-then-call-then-record is deliberately non-transactional — concurrent turns can overshoot by ~one 8192-max_tokens turn each; accepted. The used/allowance bar is the shared `CreditsBar` (`app/dashboard/creditsBar.tsx`), rendered whenever the allowance is > 0 ("resets {date}" for paid, "past 30 days" for free) on both settings and the Overview usage panel.
- **BYO OpenRouter key** (not-activated users / allowance exhausted): the permanent fallback. Entered in the "Models" section of `/dashboard/settings`, stored in the app-owned `user_secret` table (`db/setup.sql`, one row per user, `openrouter_key` column). Write-only like `auth_token`: blank keeps, checkbox clears; the client only ever sees a boolean. `saveOpenrouterKey` in `app/dashboard/(shell)/settings/actions.ts`; server-side reads via `getOpenrouterKey`/`hasOpenrouterKey` in `lib/openrouter.server.ts`. Neither key ever leaves the server; BYOK turns are never recorded in the ledger. No key and no credits → the turn errors as a value (upgrade/add-key hint).

When the user has a key set **or** is on an activated tier with a credit allowance (`limitsFor(level).modelCredits > 0` — so any activated tier today, but never level `null`), the designer toolbox lists OpenRouter's models (`listOpenrouterModels` in `lib/openrouter.server.ts` — public `GET https://openrouter.ai/api/v1/models`, deliberately unauthenticated so the 1h-revalidate Next data cache is shareable) as drag-spawnable model nodes. Each `OpenrouterModel` also carries `outputModalities` (`architecture.output_modalities` filtered to `text`/`image`), which drives the agent node's dynamic `output` select, and `supportsReasoning` (`supported_parameters` includes `"reasoning"`), which drives the agent node's dynamic `reasoning` select.

### Workflows

Users create event-driven workflows edited in a node-graph designer (new users start with the seeded inactive example workflow — see the auth section). Workflows trigger from **event nodes** (catalog category `events`): the scheduled `schedule` ("scheduled to run") node whose `config.cron` holds a 5-field cron fired by the cron tick, and the real-time **extension event nodes** keyed `event:<id>` (`event:discord-mentioned`, `event:telegram-message`) delivered by the in-process ingress transports (Discord Gateway listener, Telegram long-poller). Entry resolution is catalog-driven (`byKey[type].category === "events"`), so a new event type is one descriptor in `lib/integrations.ts` with no interpreter/designer change (the seam is realized — see the platform-extension bullets below). Actions are outbound **normal rectangular nodes** (header with the provider favicon + port rows + inline config rows — the old integration circle and `configPopover.tsx` are both gone; extension event nodes now share this rectangular shape too): flow in + flow out, plus **one value input port per config field** (same id, generated in the catalog build with `overriddenBy` auto-derived) that overrides the field's literal when connected; these paired ports render **inline on their config row's left edge** instead of a separate port row (`pairedInputIds`/`unpairedInputs` in `geometry.ts` — port rows zip only unpaired inputs with outputs, `portGeometry` anchors a paired port at its config row's vertical center) — so tokens/channel ids/messages can all be wired from upstream nodes (the interpreter merges connected port values into the config record before `callIntegration`; `message` still rides its own param). An action may also declare a single **value output** (`IntegrationAction.output` — `{id, label}`; the catalog appends it after `flowOut`): the interpreter stashes the sender's `{text}` result in `saturnResults` under that port + reports it via `onValue` (agent-`result`-style — `computeOutput` warns "read before the node ran" if read first), making read-style actions possible. Generated from the descriptors — four Discord providers: `discord-webhook` (webhook URL pasted into `config.webhookUrl` — stored plaintext in graph jsonb; the user's own graph, accepted), `discord-send-message` (bot-API `POST /api/v10/channels/{channelId}/messages` with `config.botToken` — same token as the `discord-mentioned` event node — and `config.channelId`), `discord-read-messages` (the only output-bearing action: `GET .../channels/{channelId}/messages?limit=N`, `config.count` clamped 1–100 default 20, result = compact chronological JSON array `[{id, author, bot, content, timestamp, attachments:[url]}]` on the `messages` output — no Telegram counterpart, the Telegram Bot API has no history endpoint and getUpdates is single-consumer/push-only), and `discord-typing` (no message: `config.status` on/off select; "on" fires `POST /channels/{channelId}/typing` — Discord auto-expires the indicator ~10s later, "off" is a no-op since Discord has no cancel call) — plus two Telegram providers: `telegram-send-message` (bot-API `sendMessage`, or `sendPhoto` multipart when the message is an image data URL; `config.botToken` from @BotFather — same token as the `telegram-message` event node — and `config.chatId`) and `telegram-typing` (`sendChatAction` "typing"; Telegram auto-expires it ~5s later, "off" is a no-op). Sends execute server-side in `lib/integrations.server.ts` `executeIntegration(userId, providerId, config, message)` — errors as values; the per-provider SSRF guard is an exact-host allowlist for the webhook (https + `discord.com`/`discordapp.com` host + `/api/webhooks/` path prefix, never loosen to substring checks) and a strict snowflake regex (`/^\d{17,20}$/`) on `channelId` for the bot senders (`botChannelConfig` — fixed URL base, untrusted config never shapes the fetch target); Telegram follows the same convention with `telegramConfig` — the token rides the URL *path* (`api.telegram.org/bot<token>/<method>`), so a strict charset regex (`/^\d{1,20}:[A-Za-z0-9_-]{25,64}$/`) plus a chat-id regex (numeric or `@username`) keep untrusted config from shaping the fetch target. The Discord senders share `postDiscordMessage` (content truncated to Discord's 2000-char cap): a `message` that is a `data:image/…;base64,…` data URL (e.g. an agent node's `output=image` result wired into the `message` port) is uploaded as a **multipart `files[0]` attachment** instead of text (decoded to bytes, filename ext from the mime subtype, `DISCORD_UPLOAD_LIMIT` 8 MiB backstop; `sendTelegramMessage` mirrors the same branch via `sendPhoto`, `TELEGRAM_UPLOAD_LIMIT` 10 MiB, text truncated at Telegram's 4096-char cap); `executeIntegration`'s length gate exempts image data URLs from the 4096-char `MAX_INTEGRATION_MESSAGE` cap, allowing up to `MAX_INTEGRATION_IMAGE` (4 MiB, mirrors the runner's `MAX_IMAGE_DATA_URL`) — through the interpreter's `callIntegration` hook, wired at the usual three sites (`callIntegration` action in `[id]/actions.ts` for test runs, `executeWorkflowRun` for cron/manual). Budget `MAX_INTEGRATION_CALLS` 20. `validateGraphStrict` warns on blank `requiredConfig` fields unless a value edge feeds that field's port. Integrations all carry the `integration` category, but each descriptor carries two more required fields that split **grouping** from **color**: `app` names the app (`discord`) and heads the node's toolbox subsection in the Apps group (grouped by first appearance in `INTEGRATIONS`; a provider's `label` is then the bare node name — `send webhook`, not `discord webhook`), while `section` names one of `INTEGRATION_SECTIONS` in `lib/integrations.ts` (`events`/`logic`/`data` — a subset of `NodeCategory`, enforced by `satisfies`; `discord-webhook` is `data`) and supplies only the **color**, mirroring the Blocks group — a discord webhook paints teal like the `print` node. The two ride along as `CatalogEntry.group` (which the MCP `get_catalog` tool reports as `app` for integrations, `server` for mcp entries) and `CatalogEntry.section`. Resolve colors through **`entryStyles(entry)`** (`lib/workflow.ts` → gray `MISSING_STYLES` when `entry.missing`, else `CATEGORY_STYLES[entry.section ?? entry.category]`), never by indexing `CATEGORY_STYLES` with `entry.category`, or integrations revert to orange; the "(deleted)" placeholder from `missingEntry` now renders **gray** via the dedicated `MISSING_STYLES` object (freeing `CATEGORY_STYLES.integration`'s orange to mean integration again). Every `CATEGORY_STYLES` entry also carries a full-perimeter `border` class (in its hue) alongside `borderL`, for the Phase-2 node-shape border unification. `edges.tsx` resolves the stroke once when building an `End` (`color`, not a category) since a node's category no longer determines its color. Adding a platform, action, or event (Telegram, Discord bot, Slack…) = one descriptor in `lib/integrations.ts` (+ one sender in `lib/integrations.server.ts` for an action, or a handler in the matching in-process ingress transport — `lib/gateway.server.ts` for Discord, `lib/telegram.server.ts` for Telegram — for an event); interpreter/designer/toolbox need no changes (a provider whose color category isn't in `INTEGRATION_SECTIONS` yet also needs its name added there; a new `app` value just heads a new subsection). `lib/integrations.ts` must only `import type` from `lib/workflow.ts` (workflow.ts value-imports it — a value import back would cycle at runtime).
- Extension event nodes (keys `event:<id>` via `eventNodeKey`, `EVENT_PREFIX` = `event:`; `event:discord-mentioned`, `event:telegram-message`): inbound trigger nodes, catalog category `events` so entry resolution / one-event rule / spawn guard all come free. They render as **normal rectangular nodes** (amber — they carry **no** `section`, unlike action nodes, so `entryStyles` colors them by their `events` category): no flow input (events are entry points), a flow `out` port **and a `payload` value output** carrying the event as a JSON string (wire it to `extract` to pick fields), plus **one value input port per config field** (same-id, `overriddenBy` auto-derived like integration actions) so tokens/filters can be wired from **variable/string nodes**. `discord-mentioned` config: `botToken` (required — `validateGraphStrict` warns when blank, via a requiredConfig loop mirroring the integration one incl. the port-fed skip) plus `guildId`/`channelId` optional filters; `telegram-message` config: `botToken` (required, from @BotFather) plus an optional `chatId` filter (numeric id or `@channelusername`). Event config is read **statically** by `getEventSubscriptions`, never by the interpreter: a value edge into a config port replaces the literal — a variable node resolves to its sentinel (substituted via the exported `substituteVariables`, per workflow owner, so transports only ever see plaintext tokens; a still-unresolved sentinel in a required field drops the subscription, in an optional filter it stays literal and matches nothing), a `string`/`number`/`literal` node resolves to its `config.value`, and any other (dynamic) source resolves to blank — `validateGraphStrict` warns on such edges. `saveVariable`/`deleteVariable` poke `subscriptionsChanged()` so variable-held token edits reconnect transports in ~2s. Delivery is real-time, not cron — per-provider **in-process ingress transports** started by `lib/background.server.ts` on production boot (so deploys drop and re-establish connections in seconds): the **Discord gateway** (`lib/gateway.server.ts`, hand-rolled Gateway client on `ws` — `serverExternalPackages` in `next.config.ts`) and the **Telegram long-poller** (`lib/telegram.server.ts`, plain-`fetch` `getUpdates` loops). Both read subscriptions via `getEventSubscriptions()` in `lib/events.server.ts` every 60s **plus** immediately (2s debounce) when a workflow mutation calls `subscriptionsChanged()` (poked by `saveWorkflow`, `setWorkflowActive`, `deleteWorkflow`, and the MCP `update_workflow`/`delete_workflow`/`save_graph` tools — the interval poll is the reconciliation backstop for anything missed; `onSubscriptionsChanged` is multi-listener, a Set returning per-listener unsubscribers). Subscriptions are normalized `{workflowId, nodeId, provider, event, botToken, config}` for every `active` workflow's event nodes with non-blank required config — `provider` is the owning `PlatformExtension.id` (from `ExtensionEventNode.platform`), `event` the descriptor id, and `config` the remaining non-blank fields (the transport-interpreted filters); each transport filters subscriptions to its own provider. Discord: one Gateway websocket per distinct bot token (diffed by token value; a failed subscription query keeps live sessions), matches plain @-mentions of the bot in `MESSAGE_CREATE.mentions` (skipping bot-authored messages as a loop guard; role/@everyone mentions don't count), applies each subscription's guild/channel filters, and dispatches each match to `ingestEvent()` (`lib/events.server.ts` — validate/claim/execute core, fire-and-forget so runs never block the ws message path). Fatal Gateway closes give up on that token until its value changes: 4004 bad token, 4010–4013; 4014 (Message Content intent not enabled in the dev portal) first retries once with `GUILD_MESSAGES` only — mention messages carry `content` even without the privileged intent, so such bots still work. Telegram: one `getUpdates` long-poll loop (25s hold) per distinct bot token — Telegram allows only a single getUpdates consumer per token; fires on **any** received message (DMs always; group messages per the bot's privacy mode — doc-only, no code), applies the optional `chatId` filter (matches numeric id or `@username`), advances `offset` unconditionally as the ack, skips backlog older than 5 min (no replay after downtime), 401/404 kill the token until its value changes (mirrors 4004), 409 (webhook set or a second consumer) retries slowly with a hint and never auto-calls `deleteWebhook`, 429 honors `retry_after`; bots never receive their own messages via getUpdates, so no loop guard is needed. **Events bypass the tier cron floor entirely** — every delivered event runs, with no per-workflow cooldown (`ingestEvent` only re-checks `active` atomically while stamping `last_run_at`). Each descriptor carries a `samplePayload` (canned JSON that seeds designer test runs + the extract path picker via `sampleEventPayload(nodeType)` → JSON string, threaded as `runWorkflow`'s `eventPayloads` map) and a `payloadDoc` (one-line shape for GRAPH_DOCS). The event node's `payload` output evaluates to `eventPayloads[nodeId] ?? ""` in the interpreter's events-category case.
- Test run: the topbar's test-event runner (event `<select>` + ▶ run) executes the in-memory graph client-side via `lib/interpreter.ts` `runWorkflow(graph, byKey, hooks, { entryNodeIds?, eventPayloads? })` — entries are the given ids (the selected event), else all event-category nodes; each runs its `"out"` flow concurrently (`Promise.allSettled`); `eventPayloads` (nodeId → JSON string, built by the designer from `sampleEventPayload(n.type)` over present event nodes) seeds each event node's `payload` value output so a test run of an extension event sees a sample payload. Shared with the server cron runner (pure graph-walking logic, all side effects injected as hooks `emit`/`callMcp`/`callIntegration`/`callAgent`/`onValue`/`signal`; exact flow-cycle detection per chain, step- and MCP-call-capped, per-step-local value memoization + cycle stack so concurrent branches can't clobber each other, stoppable — the button becomes ■ stop, wired through an `AbortSignal`) and streams output into the `[id]/console.tsx` bottom panel (drag-resizable via its top edge — height state lives in `designer.tsx`, in-memory only, never persisted; sticks to the bottom only while already there; `ConsoleLine` kind `"image"` — `text` is a data URL, emitted by the print node for `data:image/` values — renders as an inline `<img>`; the cron runner's `emit` persists it as a `describeImage` info placeholder `[image · mime · KB]` instead, never base64 into `workflow_run.log`). Flow fan-outs run branches concurrently (`Promise.allSettled`; the first failure aborts the run and siblings stop at their next step). The `await` node is the join barrier: the last arriving incoming flow edge continues the chain and fills `results` with a JSON array of its `values` edges (edge order); a barrier an upstream `if` diverged past warns "await never completed (k/n branches)" once the outermost fan-out settles. Standalone mcp and skill nodes are not executable — the interpreter skips them with a warning (grant chips only feed agent ports; old flow-wired MCP graphs degrade to that skip). Agent tool calls execute for real through the `callMcpTool` server action in `[id]/actions.ts` (allowlist- and `canCallTool`-checked, tokens stay server-side via `freshMcpToken` in `lib/registry.server.ts`, tool call in `lib/mcp.ts` `callTool`). Scheduled runs execute the same interpreter server-side via `lib/runner.server.ts` (see above). Test runs also report every computed value through the interpreter's `onValue` hook into a designer-held samples map (in-memory only, never persisted); the extract node's `path` field (`picker: "json-path"` on its `ConfigField`) gets a `{}` button that opens `[id]/pathPicker.tsx` — a JSON tree of the upstream node's last output where clicking a leaf writes the dot-path as one undo step.

### Hosted MCP server (`/mcp`)

External agents (e.g. Claude Code: `claude mcp add --transport http saturn <baseUrl>/mcp`; the shared "Connect an agent" card (`app/dashboard/connectAgent.tsx`, rendered on Overview and settings) lists commands for Claude Code, Codex, Gemini CLI, VS Code, and Cursor) edit and test-run the authenticated user's workflows:

- `app/mcp/route.ts` — stateless Streamable-HTTP JSON-RPC handler (`maxDuration` 300): hand-rolled like the client in `lib/mcp.ts`, plain JSON responses (no SSE, no session id, no batching), methods `initialize` (version negotiation across 2024-11-05…2025-06-18) / `ping` / `tools/list` / `tools/call`; notifications → 202; `GET`/`DELETE` → 405. `POST` is wrapped in `withMcpAuth(auth, …)` — the bearer token resolves to an `OAuthAccessToken` whose `userId` is the only identity input; a missing/invalid token 401s with the `WWW-Authenticate` resource-metadata pointer.
- OAuth discovery: the root `.well-known` routes (see Auth section) serve the AS + protected-resource metadata; the actual authorize/token/register endpoints live under the better-auth catch-all.
- `app/mcp/protocol.ts` — JSON-RPC envelopes/negotiation; `app/mcp/tools.ts` — 12 tool definitions (hand-written JSON Schemas) + `dispatchTool(userId, name, args)`: `list_workflows`, `get_workflow` (annotates every graph node with a read-only `label` resolved via the per-user byKey catalog — uuid node types are opaque and agents mislabel grants without it; labels are re-resolved each read and harmless if echoed into `save_graph`), `create_workflow`/`update_workflow` (metadata only, workflow cap via `getActivationLevels` — no cron param; the schedule is authored in the graph via `save_graph`; update also takes the boolean `active`), `delete_workflow`, `get_catalog` (static catalog + `buildUserCatalog`, minus `legacy`/`missing` entries — user registry entries sorted **first** and per-tool descriptions trimmed to their first line ≤150 chars via `trimDescription`, both so an agent-node caller's truncated view — the interpreter's `MAX_MODEL_RESULT_CHARS` — never cuts the user's own servers; full descriptions still reach the model at runtime via `buildToolDefs`), `get_docs` (the graph-authoring guide `GRAPH_DOCS`, split out of `get_catalog` for the same size reason — returned as plain text, not JSON), `validate_graph`/`save_graph` (`isWorkflowGraph` + size caps + `validateGraphStrict`; save rejects on hard errors, returns warnings), `run_workflow` (→ `executeWorkflowRun` with `trigger='manual'` — the only writer of that trigger value; real side effects, same as a scheduled run), `list_runs`, `get_run` (ownership via workflow join). Tool failures return as `isError` results, never JSON-RPC errors.

### SEO

- `lib/seo.ts` is the single source of site identity (`SITE_NAME`/`SITE_TAGLINE`/`SITE_TITLE`/`SITE_DESCRIPTION`/`GITHUB_URL`/`ORG_NAME`/`siteUrl`). It reads `BETTER_AUTH_URL` directly (fallback localhost) — never import `lib/subscription.ts`'s `baseUrl` here or in the metadata routes, it would drag auth/db/Stripe into their bundles. `BETTER_AUTH_URL` doubles as the canonical site URL.
- Root metadata + `viewport` (themeColor light/dark pair matching `globals.css`) live in `app/layout.tsx`; title template is `%s · Saturn`. `openGraph.images`/`twitter.images`/`icons`/`manifest` fields are deliberately absent — the file conventions (`app/opengraph-image.tsx`, `app/twitter-image.tsx` re-exporting it, `app/favicon.ico` — root-only multi-res 16/32/48 ICO, the primary Google-search favicon fetched at `/favicon.ico`; `app/icon.png` 512² + `app/icon1.png` 192² — numbered-suffix multiple `<link rel=icon>`s, 192 is a 48-multiple for Google's size preference; `app/apple-icon.png` 180², `app/manifest.ts`) override the metadata object. The OG image renders with satori via `ImageResponse` from `next/og` (GeistMono-Regular.ttf from node_modules; the landing divider's `≡`/`≣` glyphs are missing from that font and render as tofu — keep them out of OG art).
- Indexing policy: only `/`, `/terms`, and `/privacy` are indexed (canonicals + the only `app/sitemap.ts` entries). `/onboard` and `/activate` are meta-noindexed but stay crawlable (not in robots.txt, so the noindex is visible). All of `/dashboard/**` is noindexed by the metadata-only pass-through `app/dashboard/layout.tsx` **and** disallowed in `app/robots.ts` (along with `/api/`, `/mcp`; `/.well-known` stays crawlable for OAuth discovery).
- Landing JSON-LD (`app/(saturn)/page.tsx`, `@graph` of Organization/WebSite/SoftwareApplication) carries offer prices that must stay in sync with `TIERS` in `tierCard.tsx`.
- No em-dashes in marketing/SEO copy (meta descriptions, OG text, JSON-LD strings).

### Conventions

- Server actions are public POST endpoints — every action re-checks the session itself via `requireUser()` (see `app/(saturn)/activate/actions.ts`).
- Components colocate with the route that uses them (e.g. `app/dashboard/sidebar.tsx`); only cross-cutting code goes in `lib/`.
- Animations respect `prefers-reduced-motion` throughout (scene, moons, transitions, glows).
